# DOCUMENT 7: DATABASE SCHEMA MASTER DOCUMENT

## Buzz Duka — Local SQLite & Cloud PostgreSQL Data Design

**Document Version:** 1.0
**Product Name:** Buzz Duka
**Document Type:** Database and data design master document
**Purpose:** Define the main database tables, relationships, required fields, sync fields, audit fields, and data accuracy rules for Buzz Duka.
**Core Rule:** Buzz Duka’s database must protect real shop data, stock accuracy, profit accuracy, offline use, sync safety, and multi-tenant separation.

---

# 1. Purpose of This Document

This document defines the required data structure for Buzz Duka.

Antigravity must use this document when building:

* Local SQLite database
* Cloud PostgreSQL database
* Prisma schema
* Local repositories
* Backend APIs
* Offline sync engine
* Analytics engine
* Admin dashboard
* Tests

Buzz Duka must not store important business data only in UI state.

Every important action must create real database records.

# 2. Database Architecture

Buzz Duka uses two main databases:

```txt id="k8o6c1"
Local database: SQLite
Cloud database: PostgreSQL
```

## Local SQLite

Used by the mobile app for:

* Offline selling
* Product search
* Stock tracking
* Sale records
* Debt records
* Expense records
* Receipts
* Activity logs
* Sync queue
* Offline license state

## Cloud PostgreSQL

Used by the backend for:

* Synced business records
* Multi-tenant storage
* Subscription control
* Device control
* Admin dashboard
* Cross-device sync
* Backups
* Support visibility

The mobile app must not connect directly to PostgreSQL.

Mobile app should communicate with backend APIs.

# 3. Multi-Tenant Rule

Every business must be separated by `business_id`.

Every important business-owned table must include:

```txt id="kcw7ib"
business_id
```

This applies to:

* users
* devices
* categories
* products
* stock_movements
* sales
* sale_items
* receipts
* customers
* debts
* debt_payments
* expenses
* activity_logs
* sync_queue
* subscription records

One shop must never access another shop’s data.

All queries must filter by `business_id`.

# 4. ID Strategy

Buzz Duka is offline-first, so records may be created before they reach the server.

Each syncable record should support:

```txt id="3q0gco"
local_id
server_id
business_id
sync_status
created_at
updated_at
```

## local_id

Generated on the mobile device.

Used before sync.

## server_id

Generated by backend after successful sync.

May be null locally before sync.

## idempotency_key

Used to prevent duplicate uploads, especially for sales.

Recommended for:

* sales
* stock_movements
* receipts
* debts
* debt_payments
* expenses
* sync_queue items

# 30. Money Storage Rule

Money must be stored carefully.

Recommended approach:

```txt id="o53gf8"
Use decimal-safe handling.
Display money with 2 decimal places where needed.
Avoid floating point mistakes in important calculations.
```

Amounts include:

* Buying price
* Selling price
* Unit cost
* Stock value
* Line revenue
* Line cost
* Profit/loss
* Expense amount
* Debt amount
* Payment amount

Do not use careless rounding that corrupts profit.

# 36. Reversal Data Rules

Sale reversal must preserve original sale.

A reversal should:

* Mark sale as reversed
* Create reversal stock movement
* Exclude sale from active analytics
* Create activity log
* Preserve old sale item snapshots

Do not delete original sale to fix totals.

# 37. Data Accuracy Rules

Historical profit must come from:

```txt id="ol8g4r"
sale_items.line_profit_or_loss
```

Current expected profit must come from:

```txt id="rafmsf"
products.current_selling_price - products.average_unit_cost
```

Stock value must come from:

```txt id="fg8a95"
products.current_stock_quantity × products.average_unit_cost
```

Debt balance must come from:

```txt id="q3hmz0"
debts.original_amount - SUM(debt_payments.amount_paid)
```

Net profit must come from:

```txt id="ojml3t"
gross_profit - expenses
```

Do not use wrong sources.
